# 如何在多方推诿的僵局中通过“首问责任制”推动全链路闭环？

## 背景 (Situation)
在**版本发布前最后两轮全量测试阶段**，项目处于关键冲刺期。此时，编排服务（负责将 UI 拖拽逻辑转换为引擎 XML 定义）与计算引擎之间爆发了一个严重的不兼容故障。由于故障现象表现为“Spark 任务计算失败”，测试组第一时间找到了作为**该需求 Spark 引擎模块 Owner** 的我。

## 任务 (Task)
经排查，根因是编排层生成的 XML 参数在边缘叠加场景下丢失。由于修复该架构性失配预计需要 22 人天，且当时编排服务与计算引擎在设计定义模糊的情况下陷入推诿，谁也不愿在发布前承担大规模改动的风险。作为模块负责人，我的任务是：**秉持“首问责任制”，穿透全链路定界事实，并以“交付质量”为最高原则，推动版本决策层达成一个既能保障当前版本 DI（健康分数）过点，又能彻底消除隐患的最优解。**

## 行动 (Action)
1.  **穿透定界，锁定事实（对齐技术细节）**：我没有急于将单子转出，而是拉通编排服务的开发和模块 SE。通过 Trace 日志还原了全链路逻辑，让双方充分阐述观点。此时我们理清了：故障触发条件极苛刻，生产环境极难遇到，且万一发生，可通过手动修正 XML 定义进行临时绕过。
2.  **高层博弈，推动决策（说服 CCB 三大角色）**：在 CCB 会议上，我面对**版本总 SE（决策负责人）**、**测试 SE（风险评估人）**和 **PO（版本经理）**，针对“强行修复”与“风险对冲”进行了深度陈述：
    *   **陈述利弊**：在剩余测试轮次极少的情况下，强推 22 人天底层变动属于“质量自杀”，极易引入无法回测的新 Bug。
    *   **提出方案**：我成功说服他们：该场景应视为“当前版本暂不支持”，建议**降级问题单等级**（以保障版本交付的 DI 健康分），并将其**同步转为下版本的正式架构需求（AR）**进行深度解决。
3.  **工程化闭环（锁定技术合同）**：方案定下来后，为了防止“挂起即遗忘”，我持续**跟催编排服务 SE 和核心开发**，将大家讨论的技术细节、每一项的责任人和工作量全部固化，最终发布了**《方案实现纪要》**。
    *   **价值**：这份纪要确保了在下个版本录入需求时，有确定性的技术输入，避免了二次扯皮。

## 结果 (Result)
*   **按时高质量交付**：保证了版本顺利过点，DI 满足出口标准，且未引入高危脏代码。
*   **实现“流程正义”**：通过降级转需求的方式，既完成了交付任务，又确保了技术隐患在长期内能得到彻底解决。
*   **专业影响力提升**：在这种极限压力下，我表现出的理智决策能力和对全链路结果负责的态度，得到了版本大 SE 和 PO 的高度评价。

---

### 🛡️ 面试官追问预测 (Potential Q&A)

**Q：既然你是 Spark 引擎的 Owner，为什么你要去推动编排服务的方案纪要？**
**A**：这就是“首问责任制”的核心。虽然根因在编排侧，但问题爆发在我的模块。如果我只是简单把单子转走，不仅编排侧可能因压力大而选择打补丁，最终隐患还是会留给下个版本的计算引擎。我推动纪要落地，是为了把“口头承诺”变成“技术合同”，锁定技术方案，确保问题真正闭环，这也是对自己模块长期质量负责。

**Q：在版本发布的最后阶段，如何平衡“修 Bug”和“保进度”？**
**A**：核心是评估**“风险的确定性”**。在最后两轮测试中，引入 22 人天大规模改动的“未知风险”远大于该极端 Bug 的“已知风险”。通过“手动绕过+转下版本重构”的组合拳，我们将不可控的质量风险转化为了可控的需求管理，这是最理性的交付决策。
