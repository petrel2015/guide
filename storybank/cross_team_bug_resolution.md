# 如何处理跨团队需求不兼容导致的复杂架构冲突？

## 背景 (Situation)
在**版本发布前最后两轮全量测试阶段**，项目处于封网前的关键期。计算引擎、编排服务与加载入库模块之间爆发了严重不兼容故障。
**业务逻辑**：编排服务负责将用户前端“拖拉拽”生成的 ETL 流程转换为计算引擎可执行的 XML。
**技术死结**：由于前期设计模糊，编排服务在处理“自定义路径汇聚”场景时生成的 XML 参数丢失。**最棘手的是该问题具有极强的级联效应（牵一发而动全身）**：如果编排服务为了适配计算引擎而进行局部修改，会直接破坏后置“加载模块（入库）”的路径解析协议。三方陷入了逻辑死循环。

## 任务 (Task)
彻底修复需 22 人天。在极限压力下，各组由于担心“改动一处引发全局崩溃”而互相推诿。作为该需求 **Spark 引擎模块 Owner**，虽然根因不在我，但由于现象表现在我这里，我秉持**“首问责任制”**，必须穿透全链路找出真相，并在决策层推动一个既能保住 DI 健康分顺利过点，又能彻底闭环隐患的最优解。

## 行动 (Action)
1.  **穿透定界，锁定事实**：我拉通编排服务、加载模块的开发和模块 SE。通过日志还原了全链路：证明了故障触发条件极苛刻（生产环境极难遇到），且万一发生，可通过手动修正 XML 定义进行临时绕过。这为后续的“理性对冲”提供了事实依据。
2.  **多方博弈，推动决策（说服 CCB 三大角色）**：在 CCB 会议上，我面对**版本总 SE、测试 SE 和 PO**，陈述了核心逻辑：在分析尚不充分、测试轮次仅剩两轮的情况下，强推大规模底层变动是“质量自杀”。
3.  **推动“降级转需求”策略**：我成功说服决策层：接受该场景当前暂不支持，建议**降级问题单等级（保住交付过点分数）**，并同步转为下版本的正式**架构重构需求（AR）**。
4.  **工程化闭环（锁定技术合同）**：为了防止单子改版本后被遗忘，我以首问负责人身份，持续**跟催各模块 SE 和核心开发**，将技术方案细节、责任人和工作量全部固化，正式发布了**《方案实现纪要》**，作为下版本的确定性输入。

## 结果 (Result)
*   **按时高质量交付**：保证了版本顺利过点，DI 满足出口标准，未引入任何高危脏代码。
*   **精准定界与资源聚焦**：厘清了责任边界，**避免了本模块承担非本职职责的紧急修复。这不仅保护了引擎架构纯洁性，更让团队专家资源能腾出手来，全力冲刺版本的核心目标，避免了人力在杂事上的损耗。**
*   **管理口碑**：这种“在混沌中通过技术手段降噪、通过流程确保闭环”的处理方式，得到了大 SE 和 PO 的高度评价，认为我具备了从技术专家向架构管理者转型的潜质。

---

### 🛡️ 面试官追问预测 (Potential Q&A)

**Q：既然你是 Spark 引擎的 Owner，为什么你要去推动编排服务的方案纪要？**
**A**：这就是“首问责任制”的核心。虽然根因在编排侧，但问题爆发在我的模块。如果我只是简单转单，编排侧可能因压力大而选择打补丁，最终隐患还是会留给下个版本的计算引擎。我推动纪要落地，是为了把“口头承诺”变成“技术合同”，锁定方案细节，这是对自己模块长期质量负责，也是对整体流程正义的坚持。

**Q：为什么你觉得这时候不该改代码，哪怕它是个 P0 Bug？**
**A**：因为在发布前最后两周，**“未经验证的变更”是最大的质量隐患**。考虑到该场景极低概率触发且有手动绕过路径（Workaround），转为正式需求反而能换取更充分的设计与测试时间。我必须在“单点完美”和“全盘稳定”之间选择后者，这是最理性的交付决策。
