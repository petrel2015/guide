# 你们做“多业务共跑 + SparkContext 常驻”的底座，收益很大但风险也大：你如何做多租户隔离与故障止损？

### 预设回答（骨架）
- **先承认取舍**：复用运行时带来利用率与起停性能收益，但核心风险是“资源争用、依赖冲突、故障放大、变更不可控”。
- **隔离分三层（讲边界）**：
  1) 资源隔离：队列/配额、并发上限、单作业资源上限、Fair Scheduler pool/权重、动态扩缩容边界。
  2) 故障隔离：超时/熔断/限流、看门狗与自动拉起、Kill-switch、按租户/按链路隔离实例（必要时拆分运行时池）。
  3) 依赖与安全隔离：作业包校验/白名单、ClassLoader 隔离或统一依赖约束、权限与审计（HDFS/元数据/敏感数据）。
- **长驻进程的“耐久性治理”**：内存泄漏/碎片化监控、GC 与堆外指标、周期性滚动重启策略、作业级清理与缓存治理，避免“越跑越慢”。
- **变更治理**：基线回归 + 灰度发布 + 可回滚；关键变更先在小流量/典型链路验证，明确影响面与止损方案。
- **用一个真实案例收尾**：讲一次你如何把“潜在风险”变成“可控边界”（例如弹性扩缩容如何做到平稳、如何避免资源回收导致抖动/失败）。

### 面试官想听到的答案（要点）
- 你有强烈的“风险与收益对称”意识：敢复用，但更懂隔离与止损。
- 你讲得出清晰边界：什么共享、什么必须隔离；故障域怎么切。
- 你有 SRE/交付思维：观测、灰度、回滚、预案、复盘机制齐全。

### 考察点
- 架构取舍与边界设计能力
- 稳定性与风险治理（隔离、止损、变更管理）
- 多租户平台工程经验（治理与安全）
- 复盘与机制化能力

### 你的最终版（根据你的回答定制）
- **总体策略**：我们把“隔离”做成分层能力：默认局点按**单租户部署**把风险边界收紧；确实需要多租时，依托 Hadoop 侧成熟的多租能力做**存储/计算硬隔离**；对最容易放大风险的历史 JAR 作业，用**准入约束 + 专属队列/资源池**把影响面控住。
- **租户与资源隔离（多租场景）**：
  - **HDFS 隔离**：不同租户使用独立工作目录与访问权限，只允许访问本租户目录下的数据与产物。
  - **队列隔离**：不同租户使用独立队列/配额（本质是计算资源隔离），避免资源争用把问题放大成全局抖动。
- **避免“一个 JAR 拖慢整个平台/队列”**：
  - **平台编排作业（主路径）**：大多数任务走平台编排，我们在编排层做前置约束与校验（例如表老化/生命周期约束、算子能力与组合约束、调度时间/窗口约束），尽量在“提交前”就把高风险/不合规的编排拦住，避免形成不可控的资源消耗。
  - **历史 JAR 作业（特例治理）**：少量因历史原因无法迁移的 JAR 作业，我们会把责任边界讲清楚（业务自保证正确性与资源使用），并通过**单独规划队列/资源池**把其与主业务作业隔离，避免影响其它任务。
- **止损体系**：我们把 JAR 类作业纳入更强的准入与运行期治理（并发/资源上限、超时与一键停用、异常资源占用告警），让平台从“靠人盯”逐步走向“靠机制控”。
