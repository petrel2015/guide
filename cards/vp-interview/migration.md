# 平台要做大做强，关键不是技术本身，而是推广与迁移：你怎么让业务从自建 SparkApp 迁到你们的底座？怎么处理阻力并量化 ROI？

### 预设回答（骨架）
- **先讲价值主张（对业务/对公司）**：业务侧更快交付、更少非业务工作；公司侧提升资源利用率、降低 TCO、提升稳定性与治理水平。
- **分层分群迁移策略**：
  - 适配优先：标准 ETL/编排类作业优先；高定制 JAR 作业分阶段治理（先隔离、再改造/替换）。
  - 分阶段推进：PoC → 试点局点/试点业务 → 批量迁移 → 收口治理（新建作业必须走平台）。
- **迁移抓手（产品化 + 工程化）**：
  - 作业模板/可复用算子、接入文档与自助工具、观测与诊断、自助回滚与止损。
  - 迁移工具链：自动化校验、兼容性检查、影子跑/双跑比对与验收。
- **治理与协同（把阻力变成规则）**：
  - 明确责任边界与 SLO/SLA；提供迁移窗口与支持机制（驻场/培训/共建）。
  - 成本核算/分摊（showback/chargeback）与配额治理，形成“用得越多、越划算、越规范”的正循环。
- **衡量成功（用数字说服 VP）**：
  - 迁移覆盖：接入业务数、作业量覆盖率、核心链路覆盖率。
  - 效益：集群降配比例/资源利用率提升、CPU-hours/Mem-hours 下降、离线超窗率下降、人均运维量下降。
  - 体验：接入周期、人效提升、工单量与故障恢复时间下降。

### 面试官想听到的答案（要点）
- 你能把“平台建设”讲成“经营一门生意”：价值主张、目标客户、迁移策略、ROI 与规模化路径。
- 你能处理阻力与博弈：规则、机制、数据、样板间，而不是靠喊口号。
- 你能把迁移做成可复制的工程：工具链、标准、验收与持续治理。

### 考察点
- 业务驱动与 ROI 思维
- 推动落地与跨团队影响力
- 产品化与规模化能力（标准、工具、治理）
- 风险控制（双跑验收、回滚、止损）

### 你的最终版（根据你的回答定制）
- **先说明边界**：这块不是我直接 owner，但我在平台侧长期和一线 SE/交付同学协作，也观察到有效的推广方法论；如果由我负责，我会按“客户痛点 → 可交付方案 → 样板间复制 → 指标闭环”来做规模化迁移。
- **获客与需求生成（从一线痛点出发）**：
  - SE 在升级/日常拜访/技术研讨中深入客户现场，拿到真实痛点（离线窗口、稳定性、成本、运维负担、交付效率）。
  - 平台侧把痛点映射为可落地的“竞争力特性/解决方案包”（例如常驻运行时降低起停损耗、队列满载提升利用率、潮汐调度降低 TCO），并把收益口径标准化（用耗时、CPU-hours、失败率等指标说话）。
- **迁移策略（降低风险、做出样板间）**：
  - 从“标准编排类 ETL”优先切入，快速做出 1-2 个标杆局点/标杆链路；沉淀迁移手册与作业模板，复制到更多局点。
  - 历史 JAR 作业按阶段治理：短期先用专属队列/资源池隔离；中长期结合约束与能力补齐逐步迁移到平台编排，减少不可控面。
- **处理阻力（让迁移变成确定性）**：
  - 用“双跑/影子跑 + 验收指标”降低正确性与稳定性担忧；出现问题能快速止损（特性开关/回退路径/隔离队列）。
  - 把“平台承诺”和“业务自担”边界讲清楚：平台提供 SLA、诊断与治理；特殊作业有明确准入与隔离策略。
- **ROI 量化（VP最关心）**：
  - 覆盖：接入业务数、作业量覆盖率、核心链路覆盖率。
  - 效益：端到端耗时与离线超窗率、CPU-hours/Mem-hours、队列利用率/满载率、集群降配比例。
  - 人效：接入周期、人均运维量、工单量与 MTTR。
- **我可以贡献的能力**：我会把平台侧的性能/成本收益口径做成标准基线与报告模板，让 SE 推广时“可讲、可证、可复制”，降低迁移沟通成本。
