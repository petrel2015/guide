# 讲一次你推动解决的“重大问题”（性能劣化/稳定性事故/交付危机）：你怎么止损、怎么定位、怎么跨团队推动、怎么复盘沉淀？

### 预设回答（骨架）
- **影响面与止损（先保业务）**：影响了哪些业务/链路？是否触发 SLA/离线超窗？你第一时间做了什么止损动作（限流/降级/隔离队列/回滚/开关），为什么这么做？
- **定位路径（数据驱动）**：用哪些事实快速缩小范围（基线对比、关键耗时分解、event log、GC/IO/Shuffle 指标、火焰图/JFR 等）？怎么证明根因而不是猜测？
- **关键决策（取舍与边界）**：你在哪些点做了取舍（性能 vs 稳定、短期止血 vs 长期根治、局部修复 vs 架构调整）？决策依据是什么？
- **跨团队推进（Owner 能力）**：如何组织协作（War room、分工、节奏、升级汇报）？碰到阻力/扯皮怎么解决？
- **结果与收益（量化）**：故障恢复用时（MTTR）、指标回归情况、避免复发的收益。
- **复盘沉淀（机制化）**：沉淀了哪些机制（监控/告警、回归门禁、规范/模板、自动诊断、开关与止损手册），确保同类问题不再靠人盯。

### 面试官想听到的答案（要点）
- 你是“对结果负责”的 Owner：先止损，再定位，再根治，最后机制化。
- 你用数据做判断，并能在压力下做取舍与沟通升级。
- 你能把一次事故变成长期能力（体系、工具、规范、流程）。

### 考察点
- 复杂问题的结构化与根因分析能力
- 稳定性治理与应急处置能力（止损/回滚/预案）
- 跨团队推动与影响力（Owner、沟通、协作）
- 复盘与机制化落地能力

### 你的最终版（根据你的回答定制）
- **背景与影响（发布门禁触发）**：版本发布前做性能测试，对比基线出现明显劣化；如果不解决会直接影响版本按期交付与客户侧离线窗口。
- **止损动作**：第一时间把问题定性为“发布阻断项”，推动按基线结果卡住发布节奏；并把排查目标聚焦到“基线前后差异”，避免团队在猜测里发散。
- **定位路径（先排除常见项，再找异常链路）**：
  - 先看 Spark event log：对比 GC 时间、序列化时间、OOM/spill/shuffle 等指标，没有明显差异；也没有某个 stage 明显拖慢。
  - 进一步发现异常不在执行阶段，而是**生成 job 之前卡很久**（driver 初始化/提交前链路）。
  - 用 JFR + 火焰图做对比剖析，把耗时精确定位到具体调用与代码路径。
- **根因与修复（两个关键点）**：
  1) **上游 SDK XML 解析的 debug 日志实现缺陷**：即使未开启 debug，也会先做字符串拼接再丢给 logger，导致大量无效字符串构建耗时。推动 SDK 责任人加 `isDebug`/条件判断，避免非 debug 场景仍做拼接。
  2) **YARN → K8s 行为差异导致 driver 启动资源不足**：driver 配置的 requests/规格一致，但 K8s 的 limit 更严格、无法“偷资源”，而 YARN 场景存在一定弹性；因此在 K8s 上 driver 初始化 CPU/内存被限制，启动阶段负载高、耗时拉长。我们调整 K8s 的资源 limit（弹性上限）以对齐 YARN 的可用资源弹性。
- **结果**：driver 初始化明显加速，调度 `scheduler delay` 明显缩短，最终性能回归到与基线一致，保障版本如期发布并保持了计划交付节奏。
- **跨团队推进**：该问题同时涉及 SDK 与运行环境（K8s）配置，我负责把“现象 → 证据 → 根因”讲清楚并推动各责任方按优先级落地修复，避免互相甩锅。
- **复盘沉淀**：
  - 把“提交前耗时/driver 初始化耗时”纳入性能基线与回归观测项，避免只盯执行阶段指标。
  - 形成规范：日志打印必须做等级判断，避免无效字符串拼接；同时沉淀 YARN/K8s 的资源语义差异与推荐配置，减少迁移类回归。
