# 用 3 分钟讲一个你最有代表性的 Spark/数据平台项目：你解决了什么业务问题？你怎么做决策并拿到结果？

### 预设回答（骨架）
- **一句话结论（先抛结果）**：我在 `X` 项目中作为 `技术负责人/负责人`，在 `Y` 约束下（成本/工期/稳定性/合规），把 `核心指标` 做到 `量化结果`（如 SLA `A→B`、成本 `-C%`、时延 `D→E`、规模 `F→G`）。
- **背景与痛点**：业务/用户是谁？规模（作业数、数据量、峰值并发、集群规模）？当时最大的 1-2 个瓶颈是什么（稳定性/性能/成本/交付效率/治理）？
- **目标与边界**：要达成的指标（SLA、P99、吞吐、TCO、交付周期、故障率）与不可触碰红线（数据正确性、安全、兼容性）。
- **关键决策（3 个）**：每个决策说清楚「为什么选这个而不是那个」：例如架构选型（Batch/Streaming、Spark/Flink/Trino）、资源与调度策略、存储与表格式、观测与治理、迁移路径与灰度策略。
- **落地抓手（3-5 个）**：平台化/工程化动作（指标体系 + 观测、容量模型、性能基线与回归、自动化诊断、作业模板与最佳实践、分层 SLA、应急预案）。
- **结果与影响**：量化指标 + 组织影响（推广到多少业务线/团队，减少多少人力投入，提升交付速度），以及“可复用的方法论/资产”。
- **复盘（1 句话）**：如果重来会怎么做更快/更稳/更省（体现持续改进）。

### 面试官想听到的答案（要点）
- 你讲的不是“做了很多事”，而是**对业务结果负责**：指标前后对比、收益归因、可复制。
- 你能在不确定性下做取舍：**正确性/稳定性/成本/进度**的平衡与底线。
- 你具备“平台负责人”视角：**治理、标准化、规模化推广**，而不仅是单点性能调参。
- 你能影响他人：跨团队协作、推动落地、对外对上沟通。

### 考察点
- 价值与结果导向（ROI/TCO/SLA）
- 复杂问题结构化能力（抽象、取舍、优先级）
- 技术领导力（架构判断 + 深度能落地）
- 组织协同与 Owner 意识（推动、复盘、可持续）

### 你的最终版（根据你的经历定制）
- **一句话**：我们把 Spark 从“每个业务各自开发/运维一个 SparkApp”升级为“平台化 Spark 底座（Spark-as-a-Service）”，业务只提交 ETL 逻辑（XML/JAR）和调度依赖，平台统一提供运行时与资源治理；通过**运行时常驻 + 队列资源满载 + 潮汐调度弹性扩缩容**，提升资源利用率、减少长尾浪费，并在不影响白天查询 SLA 的前提下降低整体集群成本。
- **背景与痛点**：
  - 过去每个业务一个 SparkApp：资源碎片严重、长尾任务拉低利用率；版本/依赖分散，维护与排障成本高。
  - 业务除了写 ETL，还要做资源规划、性能优化、稳定性兜底，非业务工作占比高，交付慢。
- **目标与约束**：高性能（降低起停与冷启动损耗、提升吞吐）、低成本（提升利用率、降低闲置）、易用（业务只关注 ETL）；硬约束是**数据正确性**与**白天查询队列 SLA 不劣化**。
- **关键决策（3 个）**：
  1) **作业模型平台化**：业务把 ETL 逻辑以 XML/JAR 形式提交并提前上传到 HDFS，运行时按任务描述拉取逻辑执行，实现“业务逻辑”和“执行底座”解耦。
  2) **控制面/执行面解耦**：调度模块下发任务到 Java 服务，Java 服务通过 RMI 通知我们自研的 driver；driver 再根据任务信息获取 HDFS 上的计算逻辑并执行，做到多业务共跑、统一运维与优化。
  3) **性能与成本一体化治理**：让 SparkContext/driver 常驻，通过 RMI 接收任务，减少频繁起停的损耗；同时通过固定 driver 规格、动态调整 executor 规格做资源评估与“合理占满队列”。在此基础上引入**潮汐调度**：夜间把查询队列资源借给离线 Spark，白天归还保障查询；我在其中**深度参与**了通过源码改造实现 Spark 弹性动态扩缩容，并做平稳扩缩容策略，尽量降低对在跑任务的影响。
- **结果与影响**：
  - 资源：利用率 `x% → y%` / 长尾浪费下降 `x%`，队列资源满载率提升 `xpp`。
  - 性能：起停/冷启动开销下降 `x%`（或 `A → B`），整体吞吐提升 `x%`。
  - 成本：通过潮汐调度实现集群降配/节省服务器成本 `x%`，且白天查询 SLA（如 P99/超时率）保持不变或可控。
  - 组织：接入 `x` 个业务/`x` 条链路，业务侧从“写 SparkApp + 运维调参”转为“交付 ETL 逻辑 + 配调度”，平台侧沉淀可复用能力。
- **复盘**：如果重来，我会更早把“指标看板（SLA/TCO/利用率）+ 容量模型 + 性能回归”体系化固化到平台机制里，让收益更可量化、推广更快。
