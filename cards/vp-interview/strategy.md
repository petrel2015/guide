# 如果让你负责 Spark 底座未来 12-18 个月技术路线：你会押注哪 3 个方向提升“性能/成本/易用/稳定性”的综合竞争力？为什么？如何度量？

### 预设回答（骨架）
- **先对齐目标与边界**：我们服务的核心场景与硬约束是什么（离线窗口、正确性、稳定性、成本）？你选择的方向必须能转化为业务指标（TCO、SLA、人效）。
- **三个押注方向（每个方向 3 句话）**：
  1) **方向是什么**（一句话）。
  2) **为什么是它**（机会点与 80/20，解决什么痛点）。
  3) **怎么做**（关键里程碑/依赖方/风险与止损）。
- **度量方式（结果导向）**：每个方向定义 2-3 个可量化指标（例如 CPU-hours、离线超窗率、失败率、接入周期、工单量、基线耗时等），并说明“如何归因”。
- **取舍与顺序**：为什么先做 A 再做 B？哪些不做（或延后）？体现战略判断。

### 面试官想听到的答案（要点）
- 你不是罗列技术名词，而是能把路线图和业务指标强绑定。
- 你能讲出 80/20，并能解释取舍与风险控制。
- 你能把方向落到可交付里程碑，而不是“愿景”。

### 考察点
- 技术战略与路线图能力（方向、顺序、取舍）
- 业务/成本敏感度（TCO 与 ROI）
- 结果度量与归因能力（指标体系）
- 风险治理与交付能力（里程碑、止损）

### 你的最终版（根据你的回答定制）
- **先对齐“业务重点关注什么”**：我会先把离线作业按场景分层（典型 ETL、重计算、重 Shuffle、重 IO、稳定性敏感链路），用基线与日志把瓶颈归因清楚：是**试错成本高**、**计算/IO 慢**，还是**不稳定导致重试**。在这个基础上，我会押注 3 个方向，把收益落到可量化指标上（TCO/SLA/人效）。
- **押注方向 1：降低试错成本（在线调测/交互式能力）**  
  - 为什么：很多性能与稳定性问题，本质是“定位与验证成本高”，一旦需要多轮离线跑批试错，时间与资源都浪费。  
  - 怎么做：建设在线调测能力（可结合 Spark Connect），支持小样本/影子数据调试、参数快速验证、运行期观测（SQL/Stage 指标、shuffle/spill、GC）、并通过独立队列/配额隔离避免影响生产批跑。  
  - 怎么量化：平均定位耗时/验证轮次、接入与迭代周期、问题自助定位比例、平台工单量与 MTTR。
- **押注方向 2：单位资源性能（向量化/列式与执行引擎优化）**  
  - 为什么：离线场景“高性能=低成本”，核心目标是把 `CPU-hours/Mem-hours per TB` 打下来；重头通常在 SQL/Join/聚合与 IO 路径。  
  - 怎么做：以 Top 20% 高消耗作业为抓手做基线 → POC → 灰度；引入/加强向量化与列式执行（读写、算子、代码生成/Native 加速等可选路径），并配套 Shuffle/Skew/AQE 治理与数据布局优化（小文件、分区、压缩）。  
  - 怎么量化：CPU-hours/TB、P95 端到端耗时、shuffle bytes & spill、队列吞吐与满载率。
- **押注方向 3：稳定性优先（网络与 Shuffle 可靠性/可恢复）**  
  - 为什么：不稳定带来的重试与补跑，会直接放大 TCO，也会挤压离线窗口；这类问题往往是“系统性成本黑洞”。  
  - 怎么做：针对网络抖动与 Shuffle 脆弱点做专项治理，例如引入外置/独立的 shuffle 服务能力增强可恢复性、补齐网络观测与定位（丢包/重传/连接耗尽）、把异常重试从“被动重跑”变成“可控止损”。  
  - 怎么量化：成功率、重试率、离线超窗率、故障次数与 MTTR、`scheduler delay`。
- **取舍与顺序**：先把“试错成本”和“稳定性黑洞”压下去（能最快释放人效与资源），再在 Top 消耗链路上系统引入向量化/执行优化，把节省兑现成可持续降本。
