---
question: "Gluten/Auron（向量化引擎）落地阻力大，在同程的落地策略是什么？"
tags: [ "第三轮技术面" ]
---

推动底层引擎级别（如 JVM 转向 C++ 向量化）的颠覆性升级，核心矛盾是业务方“求稳怕错”与架构方“求快求省”的博弈。作为架构师，我的落地策略是“业务无感、平滑兜底、算清账本”：

- **灰度切流与自动化双跑（破除信任壁垒）**：不搞一刀切，先从外围报表或 T+1 离线批处理入手。构建自动化的“双跑与结果比对（Diff）系统”，用真实线上数据证明：在业务代码零修改的情况下，向量化引擎产出的数据绝对 100% 正确，彻底打消业务部门“怕算错数据、怕背锅”的核心顾虑。
- **智能路由与降级兜底（守住稳定性底线）**：现阶段向量化并不能完美覆盖所有算子（如复杂的自定义 UDF 或特定格式解析）。因此，必须在网关/代理层实现智能路由：只将兼容的 SQL 切流给 Gluten/Auron，不兼容的自动走原生 Spark。更关键的是运行时容错：一旦 C++ 侧发生 Crash 或 OOM，引擎要能无缝 Fallback 到 JVM 引擎重试，做到业务侧完全无感。
- **算清账本，用 ROI 驱动标杆（自上而下推动）**：技术落地的最大说服力是成本。我会优先抓取大盘里耗时最长、消耗资源最多（Top 10%）的计算密集型任务进行定向“开小灶”优化。拿到类似“单任务提速 3 倍，CU 消耗下降 50%”的硬核收益后，以此为标杆案例向技术高管汇报，利用降本的 KPI 自上而下地推动大面积覆盖。
- **平台化封装与拥抱社区（降低使用门槛）**：业务开发人员不应该关心底层是行存还是列存。作为基础架构团队，必须将 Gluten/Auron 的复杂性隐藏在数据平台（DataWorks/SQL IDE）之下，作为一个环境开关甚至默认底座。同时，针对落地中遇到的 C++ 内存泄漏等硬骨头，紧密联合开源社区或云厂商跟进发版，坚决不让业务侧承担试错成本。
