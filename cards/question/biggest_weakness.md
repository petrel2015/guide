# 缺点回答策略：从“源码单点深挖”到“大数据全局视野”

### 面试官视角分析
如果候选人只说“我不懂业务”，在高级职位上会有风险。通过这个回答，你实际上向面试官展示了你是一个：
1. **源码级专家**（提到源码钻研）；
2. **极具洞察力的调研者**（对比了 Ray/Spark）；
3. **关注前沿动态的架构师**（提到 SPIP, CDC, Operator）。

---

### 建议回答逐字稿（含硬核细节版）

“我过去的一个缺点是**容易过度陷入‘局部技术的单点优化’。**

比如在 Spark 引擎上，我习惯于扎进源码去攻克内存管理或 Shuffle 的每一个细节，虽然这确实让我具备了深厚的底层功底，但也让我在一段时间内忽略了横向的技术演进和行业大势。我意识到，**只盯着源码看，我的思维会被眼前的工具限制住。**

所以最近一两年，我一直在刻意走出这种‘局部深挖’的习惯，转向宏观视角的调研。

**比如在调研 Ray Data 时：**
我会去思考为什么它的 **Exoshuffle** 在节点内能通过 **Zero-copy（零拷贝）** 跑得那么快，进而反思我们的 Spark 或 Celeborn 能不能引入类似的内存映射技术来压榨物理上限。但同时我也看清了它的局限性——比如它不支持 SQL，所以完全缺失了 Spark 沉淀多年的 **RBO（规则优化）** 和 **CBO（代价优化）**，这种思考能让我更客观地做技术选型。

**此外，我也在持续跟踪社区的核心动态：**
- 比如最近 **Spark Operator** 的快速迭代，我觉得这是 Spark on K8s 彻底成熟的关键标志；
- 比如 **Spark 4.0 主推的 Spark Connect**：它通过逻辑计划解耦了 Client 和 Server，不仅消灭了烦人的 SDK 版本依赖，更重要的是它让 Spark 能真正跨出 JVM 语言圈子。我注意到社区正在热烈讨论 Swift、甚至 Scala 3.x 版本的接入，这对于我们让其他业务线低门槛地“加杠杆”利用 Spark 计算力具有巨大潜力；
- 甚至包括 **Native CDC 通用语法** 和 **内置 Shuffle 容错** 等提案。

我觉得，从单纯‘看代码’转型到这种‘看生态、看博弈、看演进’的思维广度，是我最近这一阶段最重要的成长。这让我现在能站在更高维度去思考：**如何用最前沿的技术，解决最实际的业务痛点。**”
