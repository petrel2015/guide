---
question: "从 YARN+HDFS 迁移至 K8s+对象存储的‘深水区’挑战"
tags: [ "第三轮技术面" ]
---

在这场双重架构演进中，我认为最大的“深水区”挑战主要集中在以下三个方面：

- **对象存储的性能鸿沟与元数据模拟**：对象存储是扁平结构，缺乏 HDFS 的原生目录重命名（Atomic Rename）和目录列表（List）性能。在迁移中，如何通过引入高性能缓存层（如 Alluxio/JuiceFS）来解决 S3/OSS 的高延迟和最终一致性问题，避免在大规模 Shuffle 或文件 Commit 阶段导致的任务性能大幅下降，是确保“换引擎”后飞机不掉速的关键。
- **K8s 在大数据调度上的‘精细度’不足**：YARN 针对大数据场景沉淀了成熟的标签调度、队列隔离和资源抢占机制。而原生 K8s 在处理大规模 Pod 启动频率、动态资源申请（Dynamic Allocation）以及 Pod 优先级调度上的性能仍有差距。我们需要在调度层通过集成 Volcano 或 Yunikorn 等插件，深度适配大数据特有的高并发、短作业调度需求。
- **数据面与控制面的‘双轨运行’一致性**：迁移过程不是瞬间完成的。在“双轨”运行期间，如何通过统一的数据湖元数据（如 Iceberg/Hudi）实现 HDFS 与对象存储的数据互通，确保同一份作业在不同介质上的计算结果完全幂等；以及如何实现一套控制平面（Control Plane）同时管理两套底层环境，确保业务感知透明且容灾可控，这是最考验架构功底的“深水区”。
- **全链路可观测性与成本治理的重塑**：从独占服务器到容器化，从本地磁盘 IO 到对象存储 API 调用。成本模型发生了质变（如对象存储的 API 请求计费、K8s 跨 AZ 流量费）。我们需要构建全新的全链路可观测体系，实时监控“存、算、网”的指标异常，防止在迁移过程中出现不可预期的资源瓶颈或成本爆炸。
