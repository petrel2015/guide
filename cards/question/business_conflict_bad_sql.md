---
question: "业务方写烂 SQL 导致变慢，大促期间强要资源，给还是不给？"
tags: [ "第三轮技术面" ]
---

这是一道考察“大局观、应急处理与技术治理”的经典问题。作为架构/平台负责人，我的回答是：**“大局为重，先救火再追责；给资源，但要有策略地‘借’，并且必须附带技术干预。”**

- **大局观：业务底线第一，拒绝简单粗暴的“不给”**。大促期间，公司的核心目标是GMV和业务连续性。如果因为平台方卡资源导致大促开天窗，这是平台的失职。但也不能“无脑给”，因为大盘资源紧张，随意加几千核可能会挤占其他核心任务，导致大盘整体雪崩。
- **短期应急：降级让路与资源隔离（策略性地“给”）**。既然大盘极度紧张，不可能凭空变出几千核。我会立刻启动大促应急预案：一键降级或暂停全网的低优先级任务（如探索性分析、非核心T+1报表），腾出资源“借”给该业务。同时，将这个烂 SQL 调度到**独立的隔离队列（沙箱）**，设置硬性上限（Hard Limit），确保它不会因为过度消耗资源而拖垮整个大盘的其他核心链路。
- **极速优化：架构师下场“急救”，治标先治本**。平台团队不能只做资源的“搬运工”。我会立刻指派数据架构专家介入，在15-30分钟内进行极速诊断。很多烂 SQL（如严重的数据倾斜、未过滤分区的全表扫描、广播失效导致的 SortMergeJoin）只需要加几个 Hint、调整几个 Spark/Hive 参数，甚至改写两行代码，性能就能提升十倍，从根本上化解对几千核的无理需求。
- **长期治理：事后追责与防御性架构（秋后算账）**。大促结束后，必须组织业务产研进行 Case Study 复盘。从平台侧建立两道防线：第一是**事前拦截**，上线基于血缘和代价估算（CBO）的 SQL 审计卡点，太烂的 SQL 直接拦截在提交阶段；第二是**成本穿透（FinOps）**，按量计费，让业务线为其低效代码“买单”，用成本倒逼业务团队提升研发质量。
