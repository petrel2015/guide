---
question: "为什么 MySQL 索引使用 B+ 树，而不是红黑树或平衡二叉树？"
tags: [ "MySQL", "数据结构", "后端" ]
---

- **磁盘 I/O 效率**：数据库数据存储在磁盘上。红黑树是二叉树，逻辑上很深，千万级数据的高度可能达到 20 层以上，意味着查询可能需要 20 次磁盘 I/回；而 B+ 树通过多路平衡（高扇出），3-4 层即可覆盖千万级数据，极大减少了磁盘 I/O 次数。
- **磁盘预读与页大小**：数据库按“页”（通常 16KB）读取数据。B+ 树的一个节点大小通常设置为一个页的大小，一次 I/O 就能读入上百个键值对；而红黑树节点很小，一次 I/O 只能读入一个节点，空间利用率极低且无法利用磁盘预读特性。
- **范围查询性能**：B+ 树的叶子节点之间有双向链表连接，执行 `BETWEEN` 或 `>` 等范围查询时，只需在叶子层水平遍历；而红黑树执行范围查询需要进行中序遍历，在磁盘场景下这种跨层级的随机访问性能极差。
- **查询稳定性**：B+ 树的所有数据都存储在叶子节点，任何查找都要从根走到叶子，查询路径长度固定，性能非常平稳；红黑树的数据分布在所有节点，查找效率受数据在树中的深度影响，波动较大。
- **内存与磁盘的差异**：红黑树更适合“内存中”的数据结构（如 Java 的 TreeMap 或 Redis 内部某些结构），因为内存访问不存在寻道延迟；但对于涉及大量磁盘访问的数据库系统，减少树的高度（B+ 树的核心优势）才是性能优化的核心。
